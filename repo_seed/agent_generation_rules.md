# Agent Generation Rules

These rules govern how new files should be generated in the Cultural Motifs Repo.

The goal is to keep outputs specific, structured, conservative, and usable.

---

## Rule 1 — specificity beats vibes

Always prefer a specific formal, cultural, regional, material, or symbolic term over a vague aesthetic label.

### Bad
- ancient pattern
- tribal lines
- exotic motifs
- ethnic ornament
- mystic geometry

### Better
- stepped fret
- interlace knotwork
- vegetal arabesque
- lotus-and-papyrus border
- star-polygon tessellation
- feathered serpent imagery
- heraldic beast motif

If a phrase sounds like it belongs on a fake candle label, kill it.

---

## Rule 2 — separate visual form from symbolic meaning

Do not treat visual resemblance as proof of shared meaning.

Whenever possible, distinguish:

- what the motif looks like
- where it appears
- how it is composed
- what it may symbolize
- where symbolism is context-dependent or debated

A motif can look similar across traditions and still function differently.

---

## Rule 3 — do not flatten adjacent traditions

Neighboring or related traditions must not be treated as interchangeable.

### Examples
- Japanese is not just a subset of “Asian”
- Chinese and Japanese decorative systems are not identical
- Maya is not interchangeable with Aztec/Mexica
- Egyptian should not drift into fantasy “pharaoh-core”
- Islamic geometric systems are not identical to all ornamental traditions in the Islamic world
- Russian folk ornament is not the same as generic European medieval decoration

If the file starts sounding like “same shit different country,” it is wrong.

---

## Rule 4 — anti-drift is mandatory

Every region or major subject must include some version of:

- what this is
- what this is not
- common confusion risks
- likely bad outputs
- repair strategies or corrective notes

Drift-control is not optional cleanup.
It is part of the core file logic.

---

## Rule 5 — no invented certainty

Do not present uncertain symbolism as fixed truth.

Use cautious language where appropriate:

- often associated with
- commonly used in
- one recurring interpretation is
- context-dependent
- not universal across all examples
- varies by period, place, and medium

Do not bluff.

---

## Rule 6 — no tourist-shop tone

Avoid the tone of:

- gift shop plaques
- pseudo-mystical Pinterest captions
- fantasy lore wiki sludge
- “ancient secrets” AI drivel
- sacred-energy-core nonsense

The repo should sound:
- smart
- grounded
- structured
- useful

Not breathless. Not fake scholarly. Not full of incense smoke.

---

## Rule 7 — system usefulness matters

When relevant, include notes useful for:

- prompt construction
- retrieval tags
- filtering
- shader behavior
- repeat systems
- border generators
- tiling logic
- compositional layout
- procedural placement

This repo is not only for reading.
It is for building with.

---

## Rule 8 — examples must be controlled

When providing examples, prefer:

- motif families
- recurring formal structures
- compositional tendencies
- material tendencies
- confusion pairs
- structured comparisons

Avoid giant chaotic lists with no analytical frame.

---

## Rule 9 — headings must stay stable

Comparable files should use comparable section structures.

Do not randomly invent a new structure for each file type.

Stable headings make the repo:
- easier to navigate
- easier to compare
- easier to parse programmatically
- easier for agents to imitate

---

## Rule 10 — narrow beats fake breadth

If uncertain, go narrower.

It is better to write a careful file about one well-defined motif family than a broad file full of decorative mush.

Narrow + specific > broad + fake.

---

## Rule 11 — distinguish scale and context

Whenever useful, identify whether the motif typically appears as:

- icon
- emblem
- border
- field pattern
- garment detail
- architectural surface
- manuscript element
- carved relief
- ceremonial object detail
- scene element

Scale and placement matter.
They change how motifs function visually.

---

## Rule 12 — formal analysis should be concrete

Use formal language that helps visual systems and artists.

Useful categories include:
- symmetry
- repeat logic
- radial vs linear arrangement
- border logic
- density
- contour style
- abstraction level
- figure-ground behavior
- field segmentation
- layering

Avoid empty adjectives unless they are tied to actual form.

---

## Rule 13 — symbolic claims should stay proportionate

Not every file needs a heavy symbolic section.
Some files should remain mostly formal.

If symbolism is:
- thinly evidenced
- broad
- disputed
- secondary to the visual issue

then keep it short and clearly qualified.

---

## Rule 14 — cross-cultural comparison must preserve differences

Cross-cultural files should compare:

- shared forms
- divergent meanings
- differing material contexts
- differing compositional roles
- confusion risks

They must not imply:
- universal sameness
- automatic common origin
- aesthetic interchangeability

---

## Rule 15 — structured data must mirror the markdown

When generating JSON or CSV:

- preserve stable field names
- use concise values
- avoid comments
- avoid invalid syntax
- mirror the taxonomy used in the prose
- keep identifiers consistent

The structured data should support the markdown, not contradict it.

---

## Rule 16 — do not oversell “sacred”

Use the word “sacred” only where it is genuinely relevant and contextually appropriate.

Do not apply “sacred” to everything old, geometric, ornate, or ceremonial.
That kind of inflation makes the repo stupid fast.

---

## Rule 17 — prompt advice must be practical

Prompt-system files should help users specify:

- which culture or region
- which motif families
- which material context
- which compositional behavior
- which anti-drift exclusions or repairs

Prompt advice should not just say “be specific.”
It should show what to specify.

---

## Rule 18 — if exemplar files exist, obey them

When exemplar files exist, imitate them closely unless there is a strong reason not to.

Match:
- tone
- structure
- heading level
- density
- caution level
- field naming
- comparison logic

Do not freestyle just because you can.

---

## Final principle

This repository should help produce outputs that are:

- more precise
- more culturally distinct
- more visually coherent
- more machine-usable
- less generic
- less embarrassing

If a generated file increases vagueness, aesthetic flattening, or fake certainty, it has failed.
